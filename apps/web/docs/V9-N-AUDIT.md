# V9-N · Audit alignement Design OS V2

**Date** : 2026-05-14
**Référence** : apps/web/docs/DESIGN-OS-V2.md
**Scope** : 12 fichiers clés
**Statut** : Diagnostic — aucune modification de code

---

## 1. Synthèse executive

Le codebase Waimia est globalement bien aligné sur les fondations du Design OS V2 : palette terracotta/paper correctement tokenisée, trinité typographique Instrument Serif / Inter Tight / JetBrains Mono systématiquement respectée, et spacing scale CSS cohérent. Les templates éditoriaux (EssayTemplate, CaseStudyTemplate, AuthorPageTemplate) incarnent très fidèlement le spirit Editorial Tech × Swiss. Trois forces majeures : la rigueur des tokens CSS (tokens.css irréprochable), la sobriété du composant eyebrow / kicker en mono, et la discipline anti-card-grid sur les templates Mode A.

Les trois écarts critiques qui dégradent le positionnement premium sont : (1) **l'usage excessif de Instrument Serif en lede et body** sur presque tous les templates, où l'italique Serif est utilisé comme font principale des paragraphes lead (viole la règle « never turn serif into the primary UI system ») ; (2) **la multiplication des accents terracotta** sur ConversionFunnelTemplate — kicker, flèche, numéros, métriques, comparaison labels, icône FAQ, CTA hover — l'accent perd sa rareté et son impact émotionnel ; (3) **l'absence de stratégie image/photography** sur tous les templates : zéro image éditoriale présente dans les slots ou sections, remplacée par des SVG géométriques (cercles, triangles, rectangles) génériques qui ne correspondent pas aux références A24/Apple/Acne définies par le Design OS V2.

---

## 2. Méthodologie

- Lecture intégrale de chaque fichier + extraction des choix design observés dans le HTML structurel et les `<style>` scoped
- Comparaison contre 8 axes Design OS V2 :
  1. Page type adaptation (Mode A/B + ratio)
  2. Typography (hiérarchie, fonts, italic/mono usage)
  3. Color usage (accent terracotta restraint)
  4. Spacing/Rhythm (scale, white space, breathing)
  5. Image/Visual direction (SVG vs photography, restraint)
  6. Motion (restraint, performance, mobile fallback)
  7. Components (variants Mode A/B, layouts éditorial)
  8. UX/CRO (CTAs hiérarchie, conversion, mobile-first)

## 5. Recommandations stratégiques

### La typo Serif n'est pas un système body — c'est une ponctuation

L'écart le plus systémique et le plus grave dans le codebase est l'utilisation d'Instrument Serif comme font principale des paragraphes d'introduction (lede, hero-lead, aside) sur presque tous les templates. Le Design OS V2 est explicite : « never turn serif into the primary UI system — preserve rarity and impact ». Lorsque le Serif est présent dans chaque zone de texte, il perd son pouvoir émotionnel et le site commence à ressembler à une publication générique plutôt qu'à une marque éditorial-tech. Inter Tight à 17-19px avec un line-height de 1.65 est la vraie voix Waimia pour le corps du texte — sobre, précise, suisse. Réserver Instrument Serif aux H1, H2, citations et drop-caps permettra à chaque apparition d'avoir l'impact d'une ponctuation éditoriale, comme dans les références Kinfolk ou Architectural Digest.

### L'absent le plus critique : les images

Le Design OS V2 consacre une section entière à la direction image, citant des références aussi précises qu'A24, Apple, Acne Studios et Kinfolk. Ces marques partagent une caractéristique : leurs images sont art-directées, cinématiques, émotionnellement contrôlées. Le codebase Waimia actuel n'a aucune image réelle dans aucun template — uniquement des SVG géométriques décoratifs (cercles, triangles, rectangles, lignes) qui sont l'exact opposé de cette direction. Ces SVG ne sont pas inherently mauvais — ils peuvent rester comme overlay de texture très subtil (opacity 0.06-0.12) — mais ne doivent jamais être l'élément visuel principal d'une page. Sans stratégie image concrète (slot hero-image optionnel dans chaque template, photographes / art direction définis), le site restera dans une zone « generic tech » que le Design OS V2 rejette explicitement. La priorité numéro 1 de la prochaine session design devrait être de définir la banque d'images éditoriales Waimia et de câbler les slots dans les 4 templates prioritaires (Essay, CaseStudy, Hub, ConversionFunnel).

### Les couleurs non-palette fragilisent la cohérence systémique

TechnologiesDetailTemplate introduit `#2AA876` (vert) et `#3321FF` (bleu) pour différencier les couches Claude / OS / Orchestration. Ces couleurs sont absentes du DS V2 et du fichier tokens.css. EssayTemplate utilise `#6366f1` (violet) et `#f59e0b` (ambre) pour les related cards. Ces dérives, bien qu'anecdotiques individuellement, signalent un problème de discipline systémique : dès que les tokens disponibles ne couvrent pas un besoin, les développeurs inventent des couleurs ad-hoc. La règle du DS est claire : « Si un besoin n'a pas de token → stop, signaler à Simon, ne pas inventer ». Pour différencier les couches Technologies sans violer la palette, utiliser les outils disponibles : `opacity`, `border-left` hairline coloré, taille de texte différente, kicker mono avec opacité variable.

### Le terracotta est un outil rare — pas une couleur systémique

Sur ConversionFunnelTemplate, l'accent terracotta est utilisé en 7+ endroits différents sur la même page (kicker, H1 italic, H2 italic, bullets, labels de comparaison, icône FAQ, CTA hover, métriques). Quand tout est accentué, rien ne l'est. Les références du DS V2 (COS, Rimowa, Frama) utilisent leur couleur signature avec une économie chirurgicale : 1-2 touches par écran, jamais pour la décoration utilitaire. La règle pratique : pas plus de 3 points terracotta actifs simultanément dans le viewport. Pour les templates conversion (Mode B), où la lisibilité prime, les bullets, labels et icônes peuvent être en `--ink` ou `--muted` — l'accent est réservé aux moments qui méritent l'attention émotionnelle de l'utilisateur (valeur ROI, CTA principal, chiffre signature).

### Le risque si non traité : crédibilité premium en danger

Waimia se positionne contre les « generic SaaS pages », « startup clichés » et « dashboard aesthetics » — exactement ce que le Design OS V2 cherche à éviter. Sans les correctifs ci-dessus, le site risque précisément de basculer dans ces catégories : overuse du Serif sans discrimination (cliché éditorial), SVG géométriques en décoration principale (cliché tech/fintech), terracotta omniprésent (cliché agence créative), pill CTAs (cliché SaaS). Les références citées (Awwwards, CSS Design Awards) valorisent précisément les sites qui maîtrisent la contrainte typographique, la direction image et la cohérence systémique. Un chantier de 2-3 sessions focalisées sur les 15 actions ci-dessus permettrait d'atteindre le niveau de polish nécessaire pour que le site incarne réellement le positionnement premium qu'il revendique dans son propre contenu.
